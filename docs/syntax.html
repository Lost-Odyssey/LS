Types, Methods & Interfaces

The deeper half of LS: the type system (struct / interface / generics), algebraic data types and match, error handling with Option/Result, and the ownership · move · borrow model that makes LS memory-safe without a garbage collector and without named lifetimes. Closures and static reflection have their own chapters further down. Examples are verified against the compiler.

struct & methods

A struct is a product type; its methods block holds the instance and static functions. Field separators (;, ,, or a newline) are interchangeable, and a trailing one is optional.

struct Point { f64 x; f64 y }     // two fields

methods Point {
    def shifted(&self, f64 dx, f64 dy) -> Point {   // instance method (&self)
        return Point { x: self.x + dx, y: self.y + dy }
    }
    static def origin() -> Point {                  // static: no self
        return Point { x: 0.0, y: 0.0 }
    }
}

Point p = Point { x: 3.0, y: 4.0 }      // struct literal: Type { field: value, ... }
Point q = p.shifted(1.0, -1.0)         // q = (4, 3)
Point o = Point.origin()              // static call goes through the type name
Struct literals fill every field. Write Type { a: .., b: .. } with all fields, or Type {} for the zero/empty value (each field defaulted — 0, empty Vec/Str, etc.). A bare {} is the idiomatic way to start an empty container field.

self forms — how a method borrows its receiver

An instance method declares, in its first parameter slot, how it takes the receiver. This is the same &T/&!T borrow distinction as everywhere else (see Borrowing), applied to self.

Formself isCallable on
def m(&self, …)a read-only borrow; self.f = … is rejectedowned · &T · &!T
def m(&!self, …)a mutable borrow; self.f = … allowedowned · &!T
static def m(…)absent — called as Type.m(…)the type itself
def m(…) legacya *Type pseudo-pointerowned only
methods Point {
    def show(&self)          { @print(f"({self.x}, {self.y})") }   // read-only
    def shift(&!self, f64 d) { self.x = self.x + d }            // mutating
}
Point p = Point { x: 1.0, y: 2.0 }
p.show()         // owned receiver → &self ok
p.shift(10.0)    // owned receiver → &!self ok; p.x is now 11
Prefer &self/&!self. The receiver-less legacy def m() form still parses, but a method written that way can't be called on a borrow (&T/&!T), which is limiting.

Interfaces (interface + methods Type: Interface)

An interface is a named set of method signatures (LS's "trait"). A type implements one with the merged form methods Type: Interface { … } — one keyword instead of Rust's impl Trait for Type.

import std.core.str

interface Describable {
    def describe(&self) -> Str
}

struct Circle { f64 radius }

methods Circle: Describable {            // "methods Type: Interface" = implement it
    def describe(&self) -> Str { return f"Circle(r={self.radius})" }
}

// a generic function constrained on the interface — see Generic structs below
def announce(T: Describable)(T x) { @print(x.describe()) }
announce(Circle)(Circle { radius: 2.0 })   // Circle(r=2)
A plain methods Type { … } block (no :) adds inherent methods. methods Type: Interface { … } additionally records that Type satisfies Interface, so it can be used wherever a T: Interface bound is required.

Same-name methods & qualified calls disambiguation

A type may carry the same method name from more than one source — an inherent method plus one or more interfaces, or two different interfaces. They coexist; you pick one of three ways.

Inherent priority

If the type has an inherent method of that name, a bare obj.m() call always resolves to it. An interface of the same name is reachable only through a qualified call.

import std.core.str

struct Doc { Str body }
methods Doc            { def render(&self) -> Str { return self.body } }            // inherent
methods Doc: Show      { def render(&self) -> Str { return f"<{self.body}>" } }   // interface

Doc d = Doc { body: "hi" }
d.render()          // "hi"   — inherent wins, no ambiguity
Show.render(&d)     // "<hi>" — qualified call picks the interface version

Qualified call — Interface.method(recv, args…)

When two interfaces provide the same name (no inherent method to win), a bare call is a compile error; name the interface to choose. The receiver is the first argument — auto-borrowed for an &self method (bare or &recv), and written &!recv for an &!self method.

interface Source { def close(&!self) -> int }
interface Sink   { def close(&!self) -> int }
struct Pipe { int n }
methods Pipe: Source { def close(&!self) -> int { return 100 } }
methods Pipe: Sink   { def close(&!self) -> int { return 200 } }

Pipe p = Pipe { n: 0 }
p.close()           // ✗ ambiguous: interfaces 'Source' and 'Sink' both provide 'close'
Source.close(&!p)   // 100
Sink.close(&!p)     // 200
Ambiguity is rejected at compile time. A bare call to a name that has no inherent method and ≥2 interface providers fails with a clear error that names the interfaces and suggests the qualified form. There is no silent "first one wins".

Works on generic types too

A generic type can carry same-name methods the same way (it just needs an inherent methods X(T) { … } block present to host them — any inherent block will do).

interface Show3 { def tag(&self) -> Str }
interface Mark  { def tag(&self) -> Str }
struct Box(T) { T val }
methods Box(T)        { def tag(&self) -> Str { return "inherent" } }
methods Box(T): Show3 { def tag(&self) -> Str { return "show3" } }
methods(T) Box(T): Mark  { def tag(&self) -> Str { return "mark" } }

Box(int) b = Box(int) { val: 1 }
b.tag()           // "inherent"
Show3.tag(&b)     // "show3"
Mark.tag(&b)      // "mark"
Nothing changes for the common case: an interface method with a single provider — x.show(), x.hash(), it.next(), ~/clone, operators — is called bare exactly as before. Qualified calls and the ambiguity rule only come into play when a name is genuinely contended.

Operator overloading

Operators are built-in interfaces. Implement the interface and write the operator as the method name (Ruby-style def +). Comparison operators come in families — define one, get the rest for free.

InterfaceDefineYou also get
Add Sub Mul Div Remdef + - * / %—
Equaldef ==!= (auto)
Orderdef <> <= >= (auto)
struct Vec2 { f64 x; f64 y }

methods Vec2: Add   { def +(&self, &Vec2 r) -> Vec2 { return Vec2{ x: self.x+r.x, y: self.y+r.y } } }
methods Vec2: Equal { def ==(&self, &Vec2 r) -> bool { return self.x==r.x && self.y==r.y } }
methods Vec2: Order { def <(&self, &Vec2 r) -> bool { return self.x < r.x } }

Vec2 a = Vec2{ x: 1.0, y: 2.0 }
Vec2 b = Vec2{ x: 3.0, y: 4.0 }
Vec2 c = a + b      // (4, 6)   via Add
@print(a != b)      // true     != derived from Equal
@print(a >= b)      // false    >= derived from Order
These are the same interfaces @derive(Equal, Order) generates automatically (see Static Reflection). Hand-write the operator only when the default field-wise behaviour isn't what you want — e.g. the Order above compares by x alone.

Lifecycle interfaces — Destroy & Clone

Destruction and deep-copy are two built-in interfaces. A struct whose fields themselves own heap memory (Str/Vec/Map/a nested such struct) gets both generated automatically, field by field — you write nothing. Only a type holding a raw resource (a *T buffer, an object handle, an OS lock) needs a hand-written one.

struct Person { Str name; int age }
// Person owns a Str → destructor + deep-copy are synthesised. Nothing to write.

import std.sys.c as c
struct Buf { *u8 data; int n }            // owns a raw heap buffer

methods Buf: Destroy { def ~(&!self) { c.free(self.data) } }   // `~` = destructor
methods Buf: Clone   { def clone(&self) -> Buf { /* deep-copy data */ ... } }

The destructor ~(&!self) runs automatically at end of scope — you never call it by hand (an explicit x.~() / a manual destructor call is a compile error). clone is invoked by the compiler at every copy point.

Move-only safety. A type with a destructor and a raw (*T/object) field but no Clone is move-only: copying it is a compile error (cannot copy 'X' … it is move-only), never a silent shallow-copy / double-free. Implement Clone (deep-copy the buffer) to make it copyable. This is exactly why lock/guard types (Mutex, Guard, …) can't be accidentally duplicated.
Terminology. LS says destructor / ~ / "destroyed", and deep-copy / clone — the old magic __drop/__clone names are retired and rejected.

Generic structs & trait bounds

A type parameter goes in parentheses after the name: struct Box(T). The parameter is in scope across the methods block. Instantiate by naming the concrete type — Box(int), Box(Str) — and each instantiation is monomorphized (no boxing).

struct Box(T) { T value; Str label }

methods Box {                          // T is in scope here
    def get(&self) -> T { return self.value }
}

Box(int) bi = Box(int) { value: 5,  label: "k" }
Box(Str) bs = Box(Str) { value: "hi", label: "m" }

A trait bound restricts the type parameter to types that implement an interface. The syntax differs by where the parameter is declared:

// on a generic FREE FUNCTION: the bound rides in the type-param list (T: Interface)
def sum_all(T: Add)(Vec(T) xs, T zero) -> T {
    T acc = zero
    for x in xs { acc = acc + x }      // `+` is available because of T: Add
    return acc
}

// on a generic METHOD: a `where` clause after the signature
methods Box {
    def hashed(&self) -> u64 where T: Hash { return self.value.hash() }
}
Bounds are checked at monomorphization. Instantiating with a T that lacks the required interface is a clear missing-method compile error at the use site — never a silent wrong value. (A method-level type parameter that is a closure's return type, like def map(U)(…), is covered in the Closures chapter.)

Generic calls compose

Inside a generic body you can forward the surrounding type parameter to another generic callable — a generic free function or a method-level generic — and it instantiates transitively:

def singleton(T)(T x) -> Vec(T) { Vec(T) v = {}  v.push(x)  return v }

methods(T) Box(T) {
    // the outer T is passed straight through to the generic free function
    def wrap(&self) -> Vec(T) { return singleton(T)(self.v) }
}

This is how generic code reuses other generics parameterized by its own element type — forwarding T into a helper that builds, transforms, or collects values of that same type.

enum & pattern matching

An enum is a tagged union (algebraic data type): a value is exactly one of its variants, each optionally carrying a payload. match destructures it with compiler-enforced exhaustiveness — the combination that makes illegal states unrepresentable and "forgot a case" a compile error.

enum — tagged unions

Variants are written one per line (or ;/, separated). A variant may be a bare tag, or carry a typed payload. Variant names are Capitalized.

enum Shape {
    Empty                  // bare tag, no payload
    Circle(int r)          // one payload field
    Rect(int w, int h)     // several payload fields
}

Shape a = Circle(5)        // constructors are BARE — Circle(5), not Shape.Circle(5)
Shape b = Rect(3, 4)
Shape c = Empty

An enum may refer to itself in a payload — the compiler heap-boxes the recursive field automatically (and generates the destructor for the owned subtree):

enum Tree {
    Leaf
    Node(int value, Tree left, Tree right)   // self-recursive → auto-boxed
}
An enum carrying an owned payload (Str/Vec/a recursive subtree …) owns heap memory, so it is destroyed at end of scope and moved on binding — the same ownership rules as a struct with such fields (see Ownership).

match — destructure with exhaustiveness

match is an expression: each arm yields a value, and the whole match is that value. An arm pattern binds the variant's payload to fresh names. Coverage must be exhaustive — every variant, or a wildcard _.

def area(Shape s) -> int {
    match s {                       // missing a variant ⇒ compile error
        Empty      => 0
        Circle(r)  => r * r * 3        // r binds the payload
        Rect(w, h) => w * h
    }
}

def sum(Tree t) -> int {
    match t {
        Leaf          => 0
        Node(v, l, r) => v + sum(l) + sum(r)   // recurse into boxed subtrees
    }
}

OR-patterns & wildcard

Several patterns share one arm with |; _ matches anything (it also makes a non-enum match — e.g. over an int — total).

match n {
    1 | 2 | 3 => @print("small")      // OR-pattern: any of these → this arm
    _         => @print("other")      // wildcard catch-all
}

Arm bodies are an expression or a brace block; the separator between arms can be a comma or just a newline. A block arm yields its value with return only when inside a function whose value it is; as a sub-expression the last expression is the arm's value.

Ownership across a match is handled for you. Matching on an owned enum value consumes it (payloads move into the binders); matching on a borrow (&Shape, or auto-borrowed) binds payloads by reference, zero-copy, and leaves the value alive. This is one of LS's most-tested paths — you write the match, the compiler gets the drops right.

Bit-pattern match — bits[width:field]

A specialised arm pattern that slices an integer subject into bit-fields, MSB-first. Built for binary/wire-format parsing (headers, protocols). The field widths must sum to exactly the subject's bit width.

// an eCPRI-ish 32-bit header: version=1, type=0x14, payload=256
int raw = 0x10140100
match raw {
    bits[4:version][3:rsv][1:concat][8:msg_type][16:payload] => {
        // each name is bound here: version=1, msg_type=20, payload=256 …
        if concat { ... }       // a 1-bit field binds as bool
        @print(payload)          // 256
    }
    _ => { }
}
  • Match-value fields constrain instead of bind: bits[4:0x1] only matches when that 4-bit field equals 1 (else falls through to the next arm), so you can dispatch on a version/type field.
  • _ skips a field without binding: bits[4:_][28:rest].
  • OR-patterns work: bits[4:0xA][4:_] | bits[4:0x5][4:_] => ….
  • Subject types are the integer family — int / u8 / u16 / u32 / u64 / i64; a field wider than 32 bits binds as i64.
For streaming binary input (a cursor over a byte buffer, endianness, variable-width samples) reach for std.text.bytes's Reader; bits[…] is the pattern-match primitive it builds on.

Option, Result & error handling

LS has no exceptions and no null. Absence is Option(T) (Some/None); fallible results are Result(T, E) (Ok/Err). Both are ordinary enums — but they get first-class ergonomics: the try operator, the ! force-unwrap, and a family of combinators, so error handling stays terse without hiding control flow.

Option(T) / Result(T, E)

Constructors are bare (Some/None/Ok/Err); the expected type disambiguates. Consume them by match like any enum.

import std.core.str

Option(int) a = Some(42)
Option(int) b = None
Result(int, Str) r = Ok(100)
Result(int, Str) e = Err("not found")

match a {
    Some(x) => @print(x)      // 42
    None    => @print(-1)
}

def safe_div(int n, int d) -> Result(int, Str) {
    match d {
        0 => Err("divide by zero")
        _ => Ok(n / d)
    }
}

try — propagate on failure (Zig-style)

Inside a function that itself returns Result/Option, try expr unwraps the success value, or early-returns the Err/None from the enclosing function. It collapses the match-and-rethrow boilerplate to one keyword.

def parse_pos(int x) -> Result(int, Str) {
    match x { 0 => Err("zero")  _ => Ok(x) }
}

def chain(int a, int b, int c) -> Result(int, Str) {
    int x = try parse_pos(a)      // Ok → x; Err → return that Err now
    int y = try parse_pos(b)
    int z = try parse_pos(c)
    return Ok(x + y + z)
}
// chain(1,2,3) → Ok(6);  chain(1,0,3) → Err("zero")  (stops at the second try)

! — force-unwrap (assert success)

A trailing ! extracts the success value, or panics (prints a location + message, exits the process) on None/Err. Use it when failure is a bug, not a recoverable case. It's the eager complement to try's propagation.

Option(int) a = Some(42)
int x = (a)!                  // 42
int y = box.maybe()!          // directly on a call/method result
int z = get_some(5)! + 10      // inside a larger expression
A bare variable must be parenthesized: (a)!. Lexically ! and ? are greedy identifier suffixes (value! / empty? are no-arg method names), so the force-unwrap ! only fires after an expression ending in ) or ]. Write (a)! for a plain variable; f()! / v[i]! need no parentheses. An owned payload (Str/Vec/…) is moved out by !, so the source becomes dead afterward.

Combinators

For the common shapes that aren't worth a full match. The mutating ones consume the receiver (moving any owned payload out); the is_*? predicates only borrow, so the receiver stays usable.

CombinatorMeaning
.unwrap()success value, else panic
.expect(msg)like unwrap, panic message is msg
.unwrap_or(fb)success value, else the fallback fb
.unwrap_or_else(|| ..)success value, else compute the fallback lazily
.is_some?() .is_none?()Option predicates (borrow, don't consume)
.is_ok?() .is_err?()Result predicates
.map(U)(|x| ..)transform the success value → Option(U)/Result(U,E)
.and_then(U)(|x| ..)chain: the closure itself returns an Option/Result
.map_err(F)(|e| ..)transform the Err payload
.ok() .err()Result(T,E) → Option(T) / Option(E)
.ok_or(e)Option(T) → Result(T, typeof e)
Option(int) none = None
@print(none.unwrap_or(99))             // 99
@print(Some(7).is_some?())            // true  (Some(7) still usable after)

// headline: a missing Map key, recovered in one line
Map(Str, int) m = {}
m["a"] = 1
@print(m.get("a").unwrap_or(0))       // 1
@print(m.get("z").unwrap_or(-1))      // -1   (no panic, no exception)

// map / and_then take an explicit result-type arg, like vec.map(U)
Option(int) r = Some(4).map(int)(|x| x + 1)   // Some(5)

// Option → Result, then propagate with try
def pick(&Vec(Str) v, int i) -> Result(Str, Str) {
    Str s = try v.get(i).ok_or("index out of range")
    return Ok(s)
}
try, ! and these combinators are all compiler-lowered (to match), not library methods — so they reuse the exact same ownership/move/drop machinery and never leak or double-free an owned payload.

The three-tier container access model

Option underpins LS's deliberate split between loud, recoverable, and unchecked element access. Same idea for Vec and Map:

TierVecBehaviour on miss / OOB
loud (default)v[i]panics — the right default; a bad index is a bug
recoverablev.get(i)returns Option(T) — None on OOB
uncheckedv.get!(i)raw load, no check — escape hatch, UB if OOB
Vec(int) v = [10, 20, 30]
int a = v[1]                         // 20      — panics if i is out of bounds
Option(int) b = v.get(5)            // None    — recover with match / unwrap_or
int c = v.get!(0)                    // 10      — unchecked; only when i is provably valid

Map(Str, int) m = {"x": 1}
int d = m["x"]                       // 1       — panics on a missing key
Option(int) e = m.get("y")          // None    — recoverable
v.set(i, x) / m.set(k, v) (and v[i] = x / m[k] = v) are the write side; v.set! is the unchecked write. Pick the tier by whether an out-of-range index is a bug ([]), an expected condition (get), or already proven safe (get!).

Ownership & move semantics

LS is memory-safe with no garbage collector. Every value that owns heap memory has exactly one owner; when the owner goes out of scope the value is destroyed (RAII). Assigning or capturing such a value moves ownership, and the compiler statically forbids using what you moved away — so there are no use-after-free, double-free, or leaks, checked entirely at compile time. There are no named lifetimes ('a); see Borrowing for how LS gets away without them.

What owns heap — the value & move model

Types split into plain (copied freely, no tracking) and owning (a single owner, moved on transfer, destroyed at scope end):

CategoryTypesOn transfer
plainint family · f64/f32 · bool · char · *T · object · array(POD,N)copied (both usable)
owningStr · Vec(T) · Map(K,V) · any struct/enum with an owning field · a Block holding a captured envmoved (source dies)

An owning value is in one of three states. For Str these are visible as its cap, but the idea generalises to every owning type:

StateMeaningAt scope end
staticpoints at a read-only literal (a Str literal, cap==0)nothing to free
ownedholds a heap allocation it is responsible fordestroyed (freed)
movedownership already transferred awayskipped (the new owner frees)

Binding moves · by-value arguments clone

This is the distinction to internalise. Binding an owning value to a new variable transfers ownership (a move). Passing one as a by-value function argument makes an independent deep copy (a clone) — the argument stays alive.

Vec(int) a = [1, 2, 3]
Vec(int) b = a            // MOVE — b owns it now, a is dead
// @print(a.len())          // ❌ compile error: use after move

def total(Vec(int) xs) -> int {     // by-value param
    int s = 0
    for x in xs { s = s + x }
    return s
}
Vec(int) c = [1, 2, 3]
@print(total(c))           // 6 — c is CLONED into the call
@print(c.len())            // ✓ 3 — c is still alive afterward
Want to pass without copying? Borrow it — total could take &Vec(int) and read it zero-copy with the caller keeping ownership. That's the whole point of the next chapter. Use a by-value parameter only when the function genuinely needs its own copy.

Other points where ownership moves (source dies): capturing an owning value by-move into a closure, force-unwrapping an owned payload with !, and passing an owned temporary into a container (v.push("x".upper()) — an rvalue, moved in; a named variable there is cloned, not moved).

The compiler elides the clone when it can prove it's wasted. If a by-value argument is a named local's final use — nothing ever reads, borrows, or captures it again on any path — the deep copy is silently downgraded to a move (last-use elision). Semantics are unchanged, only the copy disappears; you never annotate anything. total(c) above still clones because c.len() reads c afterward. LS_NO_ELIDE=1 switches the analysis off.

The double-clone trap — copy once, not twice

The two rules above interact in a way that bites hard. Binding clones-on-read out of a place (an index, a field), and a by-value argument clones again. So the innocent two-step "read an element, then hand it on" makes two deep copies:

// ❌ TWO deep clones per element
T e = self.data[i]      // clone #1 — clone-on-read out of the buffer
out.push(e)              // clone #2 — e is a named local, cloned into the by-value arg

On flat data that is a harmless 2× constant. On a nested owning structure it compounds: cloning a depth-d value clones its child twice, and each of those clones clones its child twice, all the way down —

clone Wrap(depth 3)
└─ clone its Vec → clone the one child TWICE
   ├─ clone#1 (depth 2) → clone its Vec → child TWICE → depth1 ×2 → leaf ×2
   └─ clone#2 (depth 2) → clone its Vec → child TWICE → depth1 ×2 → leaf ×2

   Cost(d) = 2 · Cost(d−1)   ⟹   Cost(d) = O(2^d)
   the deepest node is copied 2^depth times

A perfectly legal 100-deep nested Vec (or JSON array — its payload is a Vec) then costs 2¹⁰⁰ clones and never returns. This was a real bug in Vec.copy; a fuzzer found it as a hang on deep-but-valid input.

The compiler now defuses this pattern automatically when the hand-off is the local's last use. The by-value clone of a named local whose final use it is gets downgraded to a move (last-use elision, see the note in the previous section) — so the ❌ snippet above actually compiles to one clone per element, and the O(2^d) blow-up cannot happen through it. The explicit spellings below are still worth knowing: they are guaranteed regardless of the analysis, and they cover the cases it excludes — the local is read again later, is carried across loop iterations, is captured by a closure, or is a by-value parameter being forwarded onward.

Guaranteed fix — clone exactly once. Don't materialise the intermediate named local. Use @dup (the explicit single-copy primitive: POD bit-copy / owning deep-clone, source untouched) — its fresh rvalue then moves into push instead of being re-cloned. Passing the place straight into the by-value slot is equivalent:

// ✓ ONE clone per element — both spellings are O(depth)
out.push(@dup(self.data[i]))   // explicit: dup once; the rvalue moves into push
out.push(self.data[i])         // implicit: the place is cloned once into the arg
@move does NOT solve this in generic code. @move(e) would turn clone #2 into a move — but only for owning element types. @move is a compile error on a POD (int/f64/ bool): a plain value has no ownership to transfer. A generic container method like Vec(T).copy is instantiated for POD and owning T, so it must use @dup / a place — those degrade to a bit-copy for POD and a deep-clone for owning T, compiling for every T. Reach for @move only to consume one specific named owning variable, never for generic per-element copying.

Use-after-move is a compile error

Once a variable is moved away it is dead — any later read or write is rejected. The state is monotonic: a move on any path (even one branch of an if) marks the variable moved for everything after.

Str s = "hello".upper()      // owned
Str t = s                     // move s → t
@print(s)                      // ❌ compile error: use after move

Str u = "hi".upper()
if cond { Str w = u }         // u is MAYBE-moved here…
@print(u)                      // ❌ still rejected — moved on at least one path
The progression is LIVE → MAYBE_MOVED → MOVED, one-way. No control-flow fixed-point is needed: if any path could have moved the value, it's treated as gone. This is conservative but predictable — and it's what guarantees no double-free.

Destructors run automatically (RAII)

An owning value is destroyed at the end of its scope — its destructor runs, freeing the heap it holds, recursively for owning fields. You never write a free; the owner that is still live at scope exit cleans up. (Globals are destroyed at program exit.)

def demo() {
    Vec(Str) names = ["ann", "bob"]   // owns a heap buffer + two owned Str
    names.push("cat".upper())
    // … use names …
}                                  // names destroyed here: each Str freed, then the buffer

Custom destructors and deep-copy hooks for types holding raw resources are the Destroy / Clone interfaces. For everything built from Str/Vec/Map/nested structs, both are generated — RAII is fully automatic.

Borrowing & lifetimes-lite

A borrow lets a function touch a value without taking ownership — zero-copy, and the caller keeps the value alive. &T is a read-only borrow, &!T a mutable one. LS has no named lifetimes: a borrow is argument-only and cannot escape, which is exactly the invariant Rust's 'a enforces — LS gets it from a lightweight inferred escape analysis instead.

&T read-only · &!T mutable

A borrow parameter reads (and, for &!T, writes) through a pointer to the caller's value. A read-only borrow can be passed implicitly (auto-borrow); a mutable borrow must be marked &!x at the call site, so mutation is always visible there.

def sum(&Vec(int) v) -> int {       // read-only borrow
    int s = 0
    for x in v { s = s + x }
    return s
}
def grow(&!Vec(int) v) {           // mutable borrow
    v.push(99)
}

Vec(int) v = [3, 5]
@print(sum(v))        // 8  — auto-borrow: no & needed for read-only
grow(&!v)            // mutation MUST be explicit: &!v
@print(v.len())       // 3  — v is still owned by the caller
  • Borrowable types: Str, Vec, Map, any struct/enum — all pass as a pointer.
  • &!T downgrades to &T (a mutable borrow satisfies a read-only parameter); the reverse is rejected.
  • The same variable can't be borrowed twice in one call (f(&!x, x) is an aliasing error).
  • On self, these are the &self / &!self method forms.

The non-escaping rule — why LS needs no lifetimes

A borrow is argument-only. It lives for the duration of a call and cannot outlive what it points at. The compiler rejects every way a borrow could escape:

  • storing a borrow in a variable that outlives the borrowed value;
  • putting a borrow in a struct field or an enum payload;
  • capturing a borrow in a closure;
  • returning a borrow of a local (only elision of an input borrow is allowed).
Because a borrow physically cannot outlive its referent, there is nothing for a named lifetime to express — this non-escaping rule is the lifetime system. It's an inferred, conservative escape analysis, not Rust's explicit 'a annotations. The trade-off: a few advanced patterns (a borrow stored in a struct, a view returned from multiple inputs) aren't expressible; LS offers value-plus-offset views (e.g. StrSlice) for those instead.

Field & element borrows — &obj.field, &v[i]

You can borrow a single field or element and hand it to a &T parameter, zero-copy, while the container stays alive. The mutable form &!obj.field lends a writable field (the mechanism behind Guard.lock).

def slen(&Str s) -> int { return s.len() }

struct Person { Str name; int age }
Person p = Person { name: "ann", age: 7 }
@print(slen(&p.name))         // borrow the field, zero-copy; p stays alive

Vec(Str) names = ["alpha", "be"]
@print(slen(&names[0]))        // borrow an element, zero-copy → 5

Borrowing for-in — for x in &v

Iterating for x in v over an owning Vec clones each element. for x in &v binds each element as a &T borrow instead — zero-copy reads, source untouched.

Vec(Str) names = ["ann", "bob"]
i64 total = 0
for x in &names {            // x : &Str — borrowed each iteration
    total = total + x.len() as i64   // auto-deref: use x like a value
}
// names is still alive here (no elements were moved or cloned)
Reads auto-deref to a value; passing the borrowed x into a parameter that wants an owned T (e.g. out.push(x)) makes an independent clone — safe, never a dangling pointer. v1 covers struct elements (Str included).

Borrowed slices — &array(T) and v[a..b]

A borrowed slice is a {ptr, len} fat pointer over a contiguous Vec(T) range. Make one with v[a..b] (half-open); its type is &array(T) (read-only) or &!array(T) (writable). The spelling reuses & + array(T) — C+Ruby flavour, no Rust &[T].

def sum(&array(int) s) -> int {     // reusable over any sub-range, zero-copy
    int t = 0
    for x in s { t = t + x }
    return t
}
def double_all(&!array(int) s) {     // writable slice: writes hit the underlying Vec
    for i in 0..s.len() { s[i] = s[i] * 2 }
}

Vec(int) v = [10, 20, 30, 40, 50]
@print(sum(v[1..4]))            // elements [1,4) → 20+30+40 = 90
@print(v[1..4].len())           // 3
&array(int) sl = v[0..2]       // bind a slice to a local (pins v while sl lives)
double_all(&!v[0..5])          // v becomes [20, 40, 60, 80, 100]
A slice is bounds-checked and governed by the escape rule (it can't outlive v). It's distinct from array(T, N), which is a fixed-size stack array, not a borrow.

Returning a borrow — single-input elision

A method/function may return a borrow, as long as it borrows exactly one input — that input's the only thing it can point at, so the lifetime is inferred (Rust calls this elision). The canonical case is borrowing a field of &self:

struct Inner { int v }
struct Box { Inner inner }

methods Box {
    def get_ref(&self) -> &Inner { return self.inner }   // borrow a field of self
}

Box b = Box { inner: Inner { v: 42 } }
&Inner r = b.get_ref()
@print(r.v)                  // 42 — no copy of Inner was made

This extends to generic methods: Vec(T)'s get_ref(&self, i) -> &T borrows an element zero-copy, so you can call a method straight on the borrowed element:

Vec(Str) words = ["alpha", "beta"]
@print(words.get_ref(0).len())       // 5 — method on the borrowed element, no clone
v1 is aggregate-only. A returned &T must point at a struct/enum (pointer ABI + field auto-deref). Returning a borrow of a POD scalar (&int) is a clear compile error — read a POD element with get! instead. Borrows as struct fields, named lifetimes, and multi-input returned borrows are deliberately not supported; use a value-offset view (StrSlice) when you need a stored window.

LS Low-level intrinsics

Four @-sigil builtins — @take, @dispose, @dup, @move — let a container author move, destroy, and copy values in raw memory the compiler doesn't track. They are the substrate the safe ownership model is built on. You rarely call them outside a hand-written container; when you do, correctness is your responsibility, not the compiler's.

The place-intrinsic model

The compiler tracks ownership only for named variables (see the move model). The moment you move values between anonymous slots of a raw buffer — a *T array, a heap box, a struct field reached by a runtime index — there is no variable to track, and the compiler's bookkeeping goes silent. The three place intrinsics operate there:

IntrinsicOperates onEffectLiveness tracked by
@take(place)a place (p[i], field, *p)move the value OUT (raw load, no clone); slot left staleyou (the container's len/metadata)
@dispose(place)samerun the value's destructor in place; buffer NOT freed; POD = no-opyou
@dup(place)samedeep-copy the value; source untouched— (produces an independent value)
@move(var)a named variable onlytransfer ownership; marks the variable movedthe compiler (use-after-move is a compile error)

They work anywhere the pattern applies, not just Vec: open-addressing Map slots, ring buffers, a Task's *T result box, a struct field handed off by a builder, broadcasting a value into a tensor.

The one-line intuition. @move is the automatic gearbox — the compiler catches your mistakes. @take/@dispose/@dup are the manual gearbox — the compiler hands you the wheel, and a wrong metadata invariant is undefined behaviour or a leak. lls run --memcheck is your only safety net.

@take · @dispose · @dup · @move

An owning value like Str is a small handle — {data, len, cap} — pointing at a heap payload elsewhere. @take is a single load of that handle: it relocates ownership in O(1) without touching the payload, where a normal read would deep-copy it.

// Vec.pop — move the last element out (no clone), then shrink len so the
// vacated slot is unreachable and never destroyed twice.
def pop(&!self) -> Option(T) {
    if self.len == 0 { return None }
    self.len = self.len - 1
    T out = @take(self.data[self.len])   // steal the handle; slot now stale but excluded by len
    return Some(out)
}

@dispose runs the destructor on the value living at a slot — freeing the payload its owning fields point to — without freeing the backing buffer. It is a no-op for POD. Use it before a raw overwrite, since a raw *T store does not drop the old occupant:

// Map.set — the slot already holds a live value; destroy it before storing the new one.
@dispose(self.vals[idx])
self.vals[idx] = v

@dup is the generic duplication primitive: a bit-copy for POD, a deep clone (via the type's Clone impl) for owning types, leaving the source live. It backs Vec.fill(x) and Map.get_or_insert.

// Vec.fill — write N independent copies of x; x itself is dropped once at scope exit.
for (int i = 0; i < self.len; i = i + 1) {
    @dispose(self.data[i])       // destroy the old element…
    self.data[i] = @dup(x)        // …then store a fresh deep copy
}

@move is the only tracked one, and the only one that takes a bare variable. Use it to force-transfer a named local's ownership — the compiler then rejects any later use:

Str a = "hi".upper()
Str b = @move(a)     // a's ownership → b; the compiler marks a dead
// @print(a)             // ❌ compile error: use after move

Misuse → UB / leaks read this

Because the place intrinsics do zero liveness tracking, every safety property is an invariant you maintain. Here is what breaks and why.

1 · @take on a named variable → silent double-free

@take does NOT mark a variable moved (only @move does). Take a named local and both it and the result own the same heap — a double-free at scope exit, with no compile error.

Str a = "hi".upper()
Str b = @take(a)     // ❌ a is NOT marked moved — a and b alias the same buffer
@print(a)             //    both freed at scope end → double-free (undefined behaviour)
Rule. To transfer a named variable, always use @move. Reserve @take for anonymous slots / boxes / fields the compiler cannot track.

2 · Raw overwrite without @dispose → leak

A raw *T store deliberately does not destroy the old occupant (the slot may be uninitialised reserved memory). Overwrite a live owning slot without disposing it first and its payload leaks:

self.data[i] = v          // ❌ if self.data[i] held a live Str, its buffer is now leaked
// correct: @dispose(self.data[i]) first, then store

3 · Reading a slot after @take / @dispose → dangling

@dispose frees the payload but leaves the handle bits in place — dangling. A later read, second @dispose, or @dup of that slot dereferences a freed pointer:

struct User { int id; Str name }

@dispose(slot)  frees only what name.data points at; the struct bytes stay:

  before                          after @dispose
  ┌──────────────┐   heap        ┌──────────────┐   heap
  │ id   = 42    │               │ id   = 42    │   ← unchanged, safe to read
  │ name.data ●──┼─▶ "alice"     │ name.data ●──┼─▶ [freed]   ← handle dangles
  │ name.len = 5 │               │ name.len = 5 │
  └──────────────┘               └──────────────┘

  read id            → SAFE   (POD, nothing was freed)
  read name's chars  → UAF    (through the dangling data pointer)
  @dispose again     → double-free   (free of a dangling pointer)
  @dup(slot)         → UAF    (clone deep-reads freed memory)

4 · Touching an uninitialised slot → free of garbage

Only the initialised range [0, len) holds real handles. A reserved-but-unwritten slot in [len, cap) is garbage; @dispose-ing it runs a destructor on a junk pointer → free(garbage) → crash. Move new values into reserved slots with a raw store, never a dispose.

5 · @take without excluding the slot → double-free

Take a value out but forget to shrink len (or overwrite the slot), and the container's destructor later drops that same, now-dangling handle a second time.

Three rules that cover all five.
  1. Every @take/@dispose must make the slot unreachable (update metadata or overwrite) before anything can read or drop it again.
  2. Every owning slot is @disposed exactly once over its life — more is a double-free, fewer is a leak.
  3. Transfer named variables with @move; use @take only for raw slots / boxes / fields.

Protocol interfaces — FromList / FromPairs

Some language sugar is opt-in per type via a small interface, exactly like the Destroy/Clone lifecycle interfaces. Implementing the interface unlocks the literal syntax; the element/key/value types come from your own generics.

InterfaceMethodUnlocks

    // implementing FromList enables the `[a, b, c]` list-literal initialiser

These replace the old reserved magic-method names (def __from_list is retired — hand-writing it is a compile error that points you at the interface). The internal wiring is unchanged; only the surface is now a named, discoverable contract.

LS Closures

LS closures are Ruby-flavoured: |x| body with a pipe-delimited parameter list. This chapter covers every closure form, the trailing-closure and type-inference sugar, capture semantics, and the sharp edges.

Overview

A closure is an anonymous function value written |params| body. It can be passed to a method (the common case), and it can capture variables from the enclosing scope. Parameter types are inferred from the context — you never annotate them.

import std.core.vec
Vec(int) v = [1, 2, 3]
v.each(|x| @print(x))              // |x| @print(x) — a closure passed to each
Vec(int) doubled = v.map(|x| x * 2)  // → [2, 4, 6]
Mental model. |x| EXPR is "one parameter x, body is EXPR, whose value is returned". For more than one statement, use a brace body |x| { ... return v }.

The two body forms

Expression body — |x| expr

A single expression; its value is the return value. No return keyword, no braces.

v.map(|x| x + 1)
v.filter(|x| x > 0)
nums.sort_by(|a, b| a - b)

Block body — |x| { ... }

Use braces for multiple statements, conditionals, or an early return. The value is produced by an explicit return.

v.map(|x| {
    int y = x * 10
    return y + 1
})

v.map(|x| {
    if x == 2 { return 999 }
    return x
})
The two forms are equivalent for a single expression: |x| x + 1 ≡ |x| { return x + 1 }. Reach for braces only when you need more than one expression.

As method arguments

The functional Vec / container methods take a closure. The element is borrowed (&T, zero-copy) into the closure; inside, you use it like a value (auto-deref).

MethodClosureReturns
v.each(f)|x| <void>nothing (side effects)
v.map(f)|x| UVec(U)
v.filter(f)|x| boolVec(T)
v.reduce(init, f)|acc, x| UU
v.any(f)/all(f)|x| boolbool
v.find(f)/pos(f)/count(f)|x| boolOption / int

Full signatures live in the stdlib reference. The map declaration, for example:

def map(U)(&self, Block(&T)->U f) -> Vec(U)
  • T — input element type, from Vec(T); inferred.
  • U — the closure's return type = result element type.

Trailing-closure sugar

When the closure is the call's last argument, it can be written as a trailing block instead of inside the parentheses — Ruby-style.

// these are equivalent:
v.each(|x| @print(x))
v.each() { |x| @print(x) }      // trailing closure (empty () then the block)

v.map(int) { |x|              // type arg in (), closure trailing
    int y = x * 10
    return y + 1
}

v.map() { |x|                  // no type arg either — U is inferred (closest to Ruby)
    int y = x * 10
    return y + 1
}
The () is required — you can't drop it entirely. The trailing { is only recognised as a closure when it directly follows a ) and starts with |. The ) is the signal "this is a call, and the { is its closure". So v.map() { |x| ... } works but v.map { |x| ... } does not.

Why the rule exists: a bare EXPR { ... } collides with block syntax everywhere — most importantly the loop body in for i in 0..100 { ... } (here the { is the body, not a closure on the range), and if/while bodies and struct literals (Point { x: 1 }). Requiring a preceding ) removes the ambiguity. If the empty () bothers you, the in-parens form v.map(|x| ...) is just as short.

For a method with more than one value argument, the others stay in the parens and the closure trails:

v.reduce(int)(0) { |acc, x| acc + x }   // init in (), closure trailing

The plain explicit form v.map(int)(|x| ...) always works too, and is the most regular — pick whichever reads best.

Type inference

For methods whose result type differs from the input (map(U), reduce(U)), U is inferred from the closure's return type — you can drop the explicit (U).

Vec(int) v = [1, 2, 3]
Vec(int) a = v.map(|x| x + 1)        // U inferred = int
Vec(Str) b = v.map(|x| f"n{x}")      // U inferred = Str
Vec(int) c = v.map(|x| {            // block body, U inferred = int
    int y = x * 10
    return y + 1
})
v1 limits. Inference covers a single method type parameter that is the return type of one closure argument (exactly the map/reduce shape). When that doesn't apply, give the type explicitly — e.g. reduce's accumulator seed sits in parens: v.reduce(int)(0, |acc, x| acc + x).
This trades a little of LS's explicit-typing convention for Ruby-style brevity. Filter/each/any/all need no type parameter at all (their result type is fixed), so they were always paren-free: v.filter(|x| x > 0).

The Block type

A closure value has type Block(ParamTypes...) -> Ret. You mostly never write it — it is the type of a closure parameter in a method/function signature, where it drives parameter-type inference for the |x| you pass.

// a function that takes a closure:
def apply_twice(Block(int)->int f, int x) -> int {
    return f(f(x))
}
apply_twice(|n| n + 1, 10)            // → 12; n inferred int from Block(int)->int
Not an inline local-variable type. You cannot declare Block()->int f = || ... as a local. Closures are written at the call site (or as a typed parameter). To hold one, pass it through a function parameter.

Capture rules

A closure may reference variables from the enclosing scope ("free variables"). How each is captured depends on its type:

Captured typeStrategyOuter variable after
int / f64 / bool / char / *Tby-copystill usable
array(POD, N)by-copy (snapshot)still usable
Str / Vec / Map / has_drop struct·enumby-movedead (use-after-move = compile error)
another Blockby-clone (deep copy of its env)still usable
int base = 10
Vec(int) r = v.map(|x| x + base)   // base captured by-copy; base still usable

Str prefix = "p-"
Vec(int) lens = v.map(|x| x + prefix.len())  // prefix MOVED into the closure;
// prefix is now dead — referencing it again is a compile error.
Nested closures are allowed: a closure body may contain another |y| .... A variable the inner closure uses from outside the outer closure is captured transitively. v1: only POD (by-copy) and Block (by-clone) may cross an outer closure boundary; a by-move type crossing two levels is rejected.

void closures (each, lock, …)

each and the concurrency guards' lock/write take a closure that returns void — it runs for side effects. An expression body is fine as long as the expression itself is void:

v.each(|x| @print(x))            // @print(x) is void → ok
g.lock(|w| w.push(1))             // push(...) is void → ok
rw.write(|b| b.nums.push(7))
A non-void value can't be silently discarded. v.each(|x| x + 1) is a type error ("expected void, got int") — computing a value and throwing it away in each is meaningless. If you want the values, use map; if you want a side effect, call a void operation (print, push, …).

Caveats / gotchas

  • Param types are never annotated. |int x| is rejected — write |x|; the type is inferred. The (int) in v.map(int) is the result type, not x's type.
  • Trailing closure needs the (). v.each() { |x| ... }, never v.each { ... } (see above).
  • Ranges are not objects. (0..100).each { ... } does not exist; iterate a range with for i in 0..100 { ... }. Closures apply to containers (Vec/Map), not ranges.
  • each borrows read-only. The element is a &T; you can't mutate it through the closure. To rewrite elements in place use indices: for i in 0..v.len() { v.set(i, v.get!(i) + 1) }.
  • by-move capture kills the source. Capturing a Str/Vec/… moves it into the closure; the outer variable is dead afterward.
  • Inference is single-type-param (v1). If a method has a type parameter that isn't a lone closure return, pass it explicitly.

Worked examples

Transform & collect

import std.core.vec
import std.core.str
def main() -> int {
    Vec(int) v = [1, 2, 3, 4]
    Vec(int) evens = v.filter(|x| x % 2 == 0)        // [2, 4]
    Vec(int) sq    = evens.map(|x| x * x)             // [4, 16]
    int total      = v.reduce(int)(0, |acc, x| acc + x)  // 10
    Vec(Str) tags = v.map(|x| f"#{x}")              // ["#1","#2","#3","#4"]
    @print(total)
    return 0
}

Capture an accumulator via a guard (concurrency-safe)

import std.core.vec
import std.sync.lock
def main() -> int {
    Guard(Vec(int)) g = {}
    g.init()
    g.lock(|w| w.push(1))                  // void expression body
    g.lock(|w| w.push(2))
    int n = g.get(int)(|w| { return w.len() })   // get(R): R explicit
    @print(n)                                // 2
    return 0
}

Closure as a function parameter (the Block type)

def twice(Block(int)->int f, int x) -> int { return f(f(x)) }
def main() -> int {
    @print(twice(|n| n * 3, 2))            // 18
    return 0
}

Modules, Numbers & FFI

The plumbing around the type system: how files compose into a program, the numeric types and their Zig-style implicit widening, and calling C through the foreign-function interface.

Modules & import

A file declares its module name with module; the entry file is module main and defines def main(). Another file pulls it in with import and calls across with a qualified mod.name.

// geometry.lls
module geometry
def area(int w, int h) -> int { return w * h }
int UNIT = 1

// main.lls
module main
import geometry
import geometry as g          // optional alias

def main() {
    @print(geometry.area(3, 4))   // qualified call → 12
    @print(geometry.UNIT)         // module global
    @print(g.area(2, 2))          // via the alias → 4
}

The standard library is a tree of modules; the import path is the directory path. The compiler resolves it under <LS_HOME>/lib/:

import std.core.vec          // lib/std/core/vec.lls — Vec(T)
import std.core.str          // Str  (auto-injected; rarely written by hand)
import std.core.map          // Map(K,V)
import std.sys.io            // file I/O
import std.sys.c as c         // C / CRT bindings, e.g. c.malloc, c.free
A type or container (Vec, Map, Str) is named directly after import — Vec(int). A module of functions (std.text.strconv, a user module) is reached through its name or alias — strconv.to_int(...). The full qualified path (std.text.strconv.to_int) also works without an alias.

Numbers & implicit widening

The numeric types are the sized-integer family, the floats, plus bool and char. Narrow values widen to wider ones implicitly (Zig-style); narrowing or any lossy conversion needs an explicit as.

Integer literals default to int; for a value past int's range (a wide bit-pattern, a CRC table) declare the wider type so the literal is read at that width. There is no implicit narrowing — it must be a visible as.

C FFI — load · extern · call

Load a shared library with load, then either declare a typed binding with extern def … from lib (type-checked) or make an untyped lib.call(...). C char* parameters take a raw *u8; pass a Str through .c_str() (which guarantees the NUL terminator).

lib msvcrt = load("msvcrt.dll")

extern def puts(*u8 s) -> int from msvcrt    // typed binding

def main() {
    Str b = "hello from FFI"
    puts(b.c_str())                      // Str → *u8 via c_str()
    msvcrt.call("puts", b.c_str())        // or an untyped dynamic call
}

Statically-linked C symbols (the CRT) are declared in an extern block with no from — this is how std.sys.c binds malloc / free / fopen:

extern {
    def malloc(i64 sz) -> *u8
    def free(*u8 p)
    def strlen(*u8 s) -> i64
}
Never pass a Str directly to an extern def. A Str is a {data, len, cap} struct with no guaranteed NUL terminator — always hand C a *u8 via .c_str(). Likewise, a pointer C malloc'd is not owned by LS; free it through the matching C function, not by letting LS take it.

Static Reflection

LS reflects types at compile time: @derive(...) asks the compiler to synthesise standard methods by walking a type's fields, and lls inspect / reflect() expose that structure. No RTTI, no runtime cost unless you opt in.

The @derive(...) attribute

Put @derive(...) on a struct or enum and the compiler generates the listed interface impls field-by-field — no boilerplate.

@derive(Hash, Equal, Order, Show)
struct Point { int x; int y }

Map(Point, int) hist = {}       // Hash + Equal → usable as a Map key
if a == b { ... }               // Equal → ==   (!= comes free)
if a < b  { ... }               // Order → <    (> <= >= come free)
@print(p.show())                 // Show  → "Point { x: 3, y: 7 }"

@ is LS's annotation sigil (cf. @time/@bench). A derive lowers to an ordinary methods Type: Interface { ... } impl that the compiler writes for you, so it composes recursively — a struct whose fields themselves derive an interface just works.

If a field can't satisfy the interface, it's a compile error at that field (e.g. @derive(Hash) on a struct with an object field). derive never recurses-and-derives fields for you; each field type must already have the interface.

enum derives use a nested match over variants (including payloads). Bare constructors mean enum variants are written Circle(5), not Shape.Circle(5). enum Order compares by variant declaration order first, then payloads; Show renders Variant(p0, p1) (or just Variant with no payload). The "—" cells (enum Serialize/Deserialize/Reflect) are not yet implemented (see caveats).

Deriving on generic structs

All seven derives work on user-defined generic structs: Equal / Hash / Order / Reflect lower to uniform operations (==, .hash(), <), and Show / Serialize / Deserialize lower a type-parameter field T to self.f.show() / self.f.to_value() / T.from_value(...). The standard library impls those interfaces for every primitive (int family, f32/f64, bool, char) and Str, so the monomorphized Box(int) / Box(Str) / Box(f64) all resolve.

@derive(Equal, Hash, Order, Reflect, Show, Serialize, Deserialize)
struct Box(T) { T value; Str label }

Box(int) a = Box(int) { value: 5, label: "k" }
if a == b { ... }                       // Equal/Order/Hash work per instantiation
Map(Box(int), int) m = {}             // Box(int) as a Map key
a.show()                                // Box { value: 5, label: k }
a.to_value().to_json()                  // {"value":5,"label":"k"}

Box(int) r = Box(int).from_value(a.to_value())   // static call, direct (no alias)
Box(Str).reflect()                       // user-type args work too
Generic gotchas.
  • No where T bound is added. The derived body just calls the operation on each field; instantiating with a T that lacks it (e.g. a plain struct that didn't derive Show put in Box(T).show()) fails at monomorphization with a clear missing-method error — never a silent wrong value.
  • Static calls on a generic instance work directly — Box(int).reflect() / Box(Str).from_value(...). (A type keyword arg is resolved by the parser; a user-type arg like Box(Str) parses as a call and the checker reinterprets it — a real make_box(cfg).render() chain is untouched. A type alias type BI = Box(int) still works too.)
  • Generic field types reflect their CONCRETE instantiated type (BI = Box(int) → value: int, BS = Box(Str) → value: Str), resolved per monomorphization via the __type_name(T) intrinsic the derive emits for type-parameter fields. Concrete (non-parameter) fields report their literal type.

Serialize / Deserialize

Serialization goes through a neutral value tree (std.core.value): the derive never knows the output format — the renderer does, so one derive feeds every format.

@derive(Serialize, Deserialize)
struct Config { Str host; int port; bool tls; Point origin }

cfg.to_value().to_json()                  // {"host":"...","port":8080,"tls":true,...}
Config back = Config.from_value(cfg.to_value())   // round-trip via the tree

// full TEXT round-trip: struct -> JSON string -> Value tree -> struct
import std.core.value as value
Config back2 = Config.from_value(value.from_json(json_text).unwrap())

// STRICT: try_from_value -> Result; a missing/mismatched field is an Err
match Config.try_from_value(value.from_json(json_text).unwrap()) {
    Ok(cfg) => { ... }
    Err(e) => { ... }      // e.g. "expected int"
}

to_value() builds the format-agnostic tree; to_json() renders it (future to_csv()/to_toml() consume the same tree). from_json(Str) -> Result(Value, Str) parses JSON text back into a tree, so a derived rebuilder reconstructs a struct straight from text. Nested structs recurse both directions.

Two rebuilders. @derive(Deserialize) synthesizes from_value -> T (BEST-EFFORT: missing/mismatched fields fall back to a zero/empty default) AND try_from_value -> Result(T, Str) (STRICT: a missing field or type mismatch is an Err, propagated). Fields must be primitive / Str / nested derived struct.

Runtime reflection — @derive(Reflect)

For programs that inspect their own types at runtime (CLI binders, debug dumps, generic tooling). opt-in — only reflected types carry the metadata.

@derive(Reflect)
struct Config { Str host; int port; bool tls }

TypeInfo ti = Config.reflect()
for fi in ti.fields { @print(fi.name + ": " + fi.type_name) }   // host: Str ...
for m  in ti.funcs  { @print(m.signature) }                  // def area(&self) -> int ...

Fields come from the struct; method signatures are scanned from the type's methods blocks (names, parameter types, return, static-ness). The method list field is funcs — methods is a keyword.

comptime field iteration

Write a generic function once and have it work over the fields of any struct — including types you don't own and can't annotate with @derive. comptime for unrolls at compile time (per concrete T), so there is no runtime cost. Where @derive gives compiler-fixed templates and runtime reflect() gives only metadata, comptime for reaches each field's value with its real static type via v.(f).

// one definition serializes ANY struct to a CSV row (zero-copy &T borrow)
def to_csv(T)(&T v) -> Str {
    Str out = ""
    comptime for f in fields(T) {
        if f.index > 0 { out = out + "," }
        out = out + f"{v.(f)}"      // v.(f): the field's value, with its real type
    }
    return out
}
to_csv(p)                            // 3,7   — T inferred from p, unrolls over Point's fields

Inside the loop the comptime handle f exposes f.name (Str), f.index (int) and f.type_name (Str) — all become literals after unroll — and v.(f) is field access by that handle. The loop above expands, for Point { int x; int y }, to out + f"{v.x}" then out + "," + f"{v.y}" — ordinary code the compiler then checks and emits (zero new codegen).

comptime if — type-directed branch pruning

A comptime if predicate is evaluated at unroll time and the branch not taken is removed before type-checking. So code that is only valid for some fields does not poison the others:

def count_flags(T)(T v) -> int {
    int n = 0
    comptime for f in fields(T) {
        comptime if f.type_name == "bool" {
            if v.(f) { n = n + 1 }     // only valid when v.(f) is bool —
        }                              // this block is dropped for Str/int fields
    }
    return n
}

Predicates compare f.name / f.index / f.type_name against literals with == != < > <= >= and combine with && || !; else comptime if chains work.

Constructing & writing — write-once builders

Beyond reading, a comptime body can construct a value of a type parameter and write its fields, so one definition can build/transform any struct. T{} (or T{a: .., b: ..}) makes a value of the type parameter; v.(f) = x writes a field (POD or owned — the old value is destroyed). And f.type is the field's concrete type, usable as a static-call receiver (f.type.from_value(x)), giving write-once deserialize with no @derive:

def from_value(T)(Value v) -> T {
    T out = T{}                          // construct a value of the type param
    comptime for f in fields(T) {
        out.(f) = f.type.from_value(obj_get(v, f.name))   // f.type = Int / Str / ...
    }
    return out
}

enum variants(T)

The enum analogue of fields(T): unroll a body once per variant, with vr.name (Str), vr.index (int) and vr.type_name (the payload type, "" when none):

comptime for vr in variants(T) {
    out = out + vr.name + " "     // e.g. "Circle Square Empty"
}

comptime match — value dispatch over the active variant

Where variants(T) is metadata-only, comptime match dispatches on the active variant of a runtime enum value and binds its payload — write-once show/serialize/visitor for any enum, no @derive. One generic arm vr(p) => expands to a real match with one arm per variant (vr exposes vr.name / vr.index / vr.has_payload / vr.payload_count; p binds the active variant's first payload):

def show_enum(T)(&T v) -> Str {
    comptime match v {
        vr(p) => {
            comptime if vr.has_payload { return f"{vr.name}({p})" }
            else { return vr.name }
        }
    }
}
// show_enum(Circle(5)) -> "Circle(5)";  show_enum(Empty) -> "Empty"

It expands to a real match v { Circle(p) => ..., Empty => ..., ... } — exhaustive by construction, reusing the existing match drop/move/checking (zero new ownership code). The no-binder form vr => is metadata-only.

  • Generic free functions infer their type args — write to_csv(p) (T is unified from the argument type), or pass it explicitly as to_csv(Point)(p). They also take &T borrow params: to_csv(p) auto-borrows, to_csv(&p) is the explicit form — zero-copy either way.
  • The predicate must be a compile-time constant over f.* and literals — a runtime value is a clear compile error. Use f.type_name == "bool" (string) for type predicates; f.type is for type position (a static-call receiver).
  • fields(T) requires a struct, variants(T) an enum — the wrong kind is a clear compile error.
  • variants(T) is metadata-only — it gives variant names/indices/payload types; to dispatch on the active variant's value use comptime match (above).

comptime constants — compile-time values & lookup tables

The other flavour of comptime: compute a value at compile time and bake it into the binary. A scalar constant is folded into a literal at every use (no runtime storage); an array constant becomes a constant-initialized global (lands in .rodata at -O2). Distinct from a plain int X = 9, which is a mutable runtime variable.

// scalar: evaluated once, folded to a literal everywhere it's used
comptime int  BITS = 9
comptime int  MASK = (1 << BITS) - 1          // 511 — can reference other comptime consts
comptime f64  INV2 = 1.0 / math.sqrt(2.0)     // math.* runs on the host at compile time

// block form: loops / if / locals, ending in `return`
comptime int FACT5 = comptime {
    int f = 1
    for i in 1..6 { f = f * i }               // 120
    return f
}

// array lookup table: 256 sin() calls run at COMPILE time, baked into .rodata
comptime array(f64, 256) SIN_LUT = comptime {
    array(f64, 256) t = {}
    for i in 0..256 { t[i] = math.sin(i as f64 * 2.0 * math.PI / 256.0) }
    return t
}
def fast_sin(int k) -> f64 { return SIN_LUT[k & 255] }   // runtime: just a load

What's evaluable (v1): int/f64/bool/char scalars and fixed array(T,N) of them; + - * / %, comparisons, && || !, bitwise/shift, as casts; references to other comptime constants; math.* constants and functions (sqrt/sin/pow/abs/…) via the host math library. Blocks add scalar/array locals, assignment, if/else, and bounded for i in 0..N.

  • Immutable. Assigning to a comptime constant is a compile error.
  • Define before use (same as ordinary globals) — a comptime constant must appear earlier in the file than the code that reads it.
  • Array element types must match exactly. Integer literals default to int; for a wider table declare it (array(i64, …) for values past int's range, e.g. a CRC32 table).
  • Not a compile-time constant? Referencing a runtime variable, an unsupported operation, a non-scalar type, or division by zero is a clear compile error — never a silent fallback to runtime.
  • Bounded. Comptime loops run under a step budget; an unbounded computation is a clear compile error, not a hung compiler.

Tooling: inspect · ir · asm

Inspect a type's fields + methods without running it — handy for exploring a type (incl. what a @derive produced, shown with tags):

$ lls inspect Point app.lls        // fields + method signatures + [static]/[Destroy] tags
ls> :methods Vec                  // in the REPL — methods only
ls> :fields Config               // fields only  (also :inspect)

See one function's optimised LLVM IR or x64 assembly — the feedback loop for hand-optimisation (e.g. checking SIMD actually got vectorised):

$ lls ir  myfn app.lls -O2          // that function's LLVM IR
$ lls asm myfn app.lls -O3 --native // its x64 asm (--native unlocks AVX)

The function name matches exactly, else by unique substring (area → Point.area); ambiguous names are listed.

Emit reusable C with Intel intrinsics for the numeric/SIMD kernel subset (scalars · pointers · Simd(T,N) · POD structs · fixed array(T,N) · control flow · __simd_*) — so an LS-authored kernel or a pointer-based model forward drops into a C/C++ codebase without the LS runtime. Simd(f32,16) → __m512, ops → _mm512_*:

$ ls emit-c kernels.lls -o kernels.c   // every in-subset fn → self-contained .c
$ clang -march=graniterapids -O3 -c kernels.c   // compiles immediately

Out-of-subset constructs (Str/Vec/Map/closures/enums/ methods) are reported with a source location and nothing is written. To pull the kernels out of a mixed model file, restrict or relax the selection:

$ ls emit-c model.lls --only sgemm,relu16 -o k.c  // just these functions
$ ls emit-c model.lls --skip-unsupported -o k.c   // all in-subset fns; warn-skip the rest

Caveats / gotchas

  • @derive(Clone) is rejected. has_drop structs are deep-copied automatically and there is no user-callable .clone() (the Clone hook is compiler-internal), so deriving it would be redundant.
  • enum derives are Equal/Hash/Order/Show. Serialize/Deserialize/Reflect on enums are not yet implemented (clear compile error if you try).
  • methods is a keyword → it can't be a field name (ti.methods won't parse). The reflected method list is ti.funcs.
  • You don't import the derive support modules — @derive(Serialize) / @derive(Reflect) inject std.core.value / std.core.reflect for you.
