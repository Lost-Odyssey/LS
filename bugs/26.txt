## Bug #26 — 循环内临时 alloca 爆栈（系统性，#24 的泛化）

### 状态：FIXED（2026-06-05）

### 现象
循环里调用任何在内部生成临时栈槽的内建方法，迭代超过 ~50000 次即 JIT 崩溃（exit 127）：
```ls
for i in 0..1000000 { string u = base.upper() }   // n>50000 → crash
```
受影响的不止 upper —— 任何 codegen 中用 `LLVMBuildAlloca(ctx->builder, ...)`（在当前
builder 位置分配）的内建：string 的 upper/lower/split/replace/substr/... 以及其它带
内部循环索引/临时槽的方法。AOT 不崩（O2 mem2reg 把循环内 alloca 提升掉）。

### 根因
LLVM `alloca` 只在**函数返回时**释放，不是 block 退出时。当 alloca 指令位于循环体所在的
basic block，每次迭代都新分配一块栈，永不回收 → 累积爆栈（JIT 默认 1MB；如 upper 的
`up.idx` i32 槽 + 临时 string，×100000 即溢出）。

bug #24 是这个家族的单点（struct 字面量 `sl.tmp`）；本 bug 是泛化——全代码库共 **62 处**
`LLVMBuildAlloca(ctx->builder, ...)` 都有同样隐患。

### 修复
新增统一 helper `cg_entry_alloca(ctx, ty, name)`（codegen.c ~L1135）：无论 builder 当前在
哪，都把 alloca 放到**当前函数的 entry block**（用 `LLVMGetBasicBlockParent(insert_block)`
动态定位函数 → entry block 首指令前插入）。entry-block alloca 每次调用只分配一次、循环内复用。

将全部 62 处 `LLVMBuildAlloca(ctx->builder, ...)` 批量替换为 `cg_entry_alloca(ctx, ...)`。
语义等价（scratch 槽的位置不影响正确性，store 仍在原位）且彻底根治整个家族。helper 自身用
局部 builder 不在替换范围。加了前向声明（首个调用点 L707 早于定义 L1135）。

注：动态大小的 `LLVMBuildArrayAlloca`（如 f-string 的 256B scratch）用不同函数名，已在
各自修复点单独置于 entry block，不在本次替换范围。

### 回归测试
`tests/test_string_loop.cmake` + `tests/samples/string_loop_test.ls`
（循环 200000 次调用 upper/lower/substr/replace/split，JIT + AOT + memcheck）。ctest 142。

### 影响范围
根治所有「循环内调用生成临时栈槽的内建方法」的 JIT 栈溢出。实际代码中循环里做字符串处理
（upper/split/replace 等）极常见，此前 n>50000 必崩，现已修复。AOT 一直不受影响。
