## Bug #24 — 循环内 struct 字面量导致 JIT 栈溢出

### 状态：FIXED（2026-06-05）

### 现象
在 JIT 模式下，循环内构造 struct 字面量超过 ~50000 次即崩溃（exit 127）：
```ls
for i in 0..100000 {
    Point p = Point { x: 1.0, y: 2.0 }   // n>50000 → crash
}
```
AOT 模式不受影响（O2 的 mem2reg pass 会提升 alloca）。
与 struct 大小、字段类型、方法调用无关——即使 struct 只有两个 f64 字段且不调任何方法也崩。

### 根因
`codegen.c:13381` 的 struct literal 临时缓冲 `sl.tmp` 使用 `LLVMBuildAlloca` 在**当前
builder 位置**（循环体内的 basic block）分配：

```llvm
foreach.body:
  %sl.tmp = alloca %Point, align 8    ;; ← 每次迭代在栈上分配 16B
```

LLVM `alloca` 仅在**函数返回时**释放（不是 block 退出时）。因此：
- 100000 次迭代 × 16 字节 = 1.6 MB 栈
- JIT 默认线程栈 = 1 MB → **栈溢出**
- AOT 编译时 LLVM O2 的 `mem2reg` pass 把循环内 alloca 提升为寄存器操作，所以不崩

### 修复
将 struct literal 的 `alloca` 放到**函数 entry block**（`codegen.c:13381`）：
```c
LLVMBasicBlockRef entry_bb = LLVMGetEntryBasicBlock(cur_fn);
LLVMBuilderRef entry_b = LLVMCreateBuilderInContext(ctx->context);
// position at entry block start
storage = LLVMBuildAlloca(entry_b, st_llvm, "sl.tmp");
LLVMDisposeBuilder(entry_b);
```

entry block 的 alloca 只分配一次（函数栈帧的一部分），循环内复用同一块内存。
这是 LLVM IR 的标准惯用法——所有其他 LS 局部变量（int/f64/string/vec）的 alloca
已经在 entry block，只有 struct literal 遗漏了。

### 回归测试
`tests/test_struct_loop.cmake` + `tests/samples/struct_loop_test.ls`
（JIT n=200000 + AOT + memcheck，之前 n=200000 必崩）。ctest 139/139。

### 影响范围
任何在循环/条件分支内使用 `StructName { field: val, ... }` 字面量的 JIT 代码。
包括但不限于：循环内创建临时 struct、向 vec(struct) push 字面量、match 臂构造 struct。
AOT 不受影响（O2 自动修正），但修复后 AOT IR 也更干净。
